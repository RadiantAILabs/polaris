# TypeScript Typegen

Polaris generates TypeScript bindings for canonical domain types via [`ts-rs`](https://github.com/Aleph-Alpha/ts-rs). The generated `.ts` files are the source of truth for downstream consumers тАФ primarily the `polaris-dashboard` Svelte app тАФ so they don't have to hand-maintain duplicates of Rust types.

This avoids the looser-DX alternative of JSON Schema via `schemars` тАФ the generated TypeScript types match the Rust structs exactly, without the schema-driven type erosion (e.g. `i64` collapsing to `number`, enums losing tag-name fidelity) that JSON Schema generators introduce.

## Output Location

Generated bindings live in `bindings/ts/src/` at the workspace root. Each derived type produces a single file named after the type (e.g. `bindings/ts/src/SessionMetadata.ts`).

The directory is checked into git. `bindings/ts/` is a self-contained npm package (`@polaris/types`) that the dashboard repo and other consumers depend on тАФ currently as a workspace path, eventually as a published artifact. `bindings/` is reserved as a cross-language root, so future Python/Go/etc. bindings can sit alongside as `bindings/python/`, `bindings/go/`, etc.

### Package layout

```
bindings/
  ts/
    package.json       # @polaris/types
    tsconfig.json
    README.md
    src/
      index.ts         # hand-maintained barrel тАФ re-exports every generated type
      SessionMetadata.ts  # generated by ts-rs
      ...
```

Consumers import from the package, not from raw paths:

```ts
import type { SessionMetadata } from "@polaris/types";
```

### Maintaining `index.ts`

`src/index.ts` is hand-maintained тАФ `ts-rs` does not produce a barrel. After adding a new `#[derive(TS)]`, regenerate (below) and add the matching `export type { ... } from "./Foo";` line. Keep entries alphabetized.

The CI `git diff --exit-code bindings/` check catches drift in the generated `.ts` files but won't catch a forgotten barrel entry, so reviewers should verify the barrel was updated whenever a new entry appears in the `Currently Derived` table.

## Workspace Feature Flag

The `typegen` feature is gated at the workspace root and is **off by default** so release builds never pull `ts-rs` into the dependency graph.

```bash
# Default build тАФ `ts-rs` is not in the dep tree.
cargo build

# Regenerate `.ts` files in bindings/.
cargo make gen-bindings
```

> **Note:** plain `cargo test --features typegen` from the workspace root does
> **not** regenerate the bindings тАФ it only runs the root package's tests, not
> the `export_bindings_*` tests inside `polaris_sessions`.
> Use `cargo make gen-bindings`, which runs those export tests across the whole
> workspace.

The feature flows from `polaris-ai` тЖТ `polaris_internal` тЖТ individual crates (currently `polaris_sessions`). Each crate that derives `TS` declares its own crate-level `typegen` feature that pulls `ts-rs` as an optional dep.

The export directory is configured globally via `TS_RS_EXPORT_DIR` in [`.cargo/config.toml`](../../.cargo/config.toml), so paths resolve to `bindings/ts/src/` regardless of which crate `cargo test` is invoked from.

## Adding `TS` Derives to a Type

Gate the derive on the `typegen` feature so default builds aren't affected:

```rust
use serde::Serialize;
#[cfg(feature = "typegen")]
use ts_rs::TS;

#[derive(Debug, Clone, Serialize)]
#[cfg_attr(feature = "typegen", derive(TS), ts(export))]
pub struct MyType {
    pub field: String,
}
```

For newtypes wrapping a borrowed type (`&'static str`), override the inner type explicitly:

```rust
#[cfg_attr(feature = "typegen", derive(TS), ts(export))]
pub struct AgentTypeId(#[cfg_attr(feature = "typegen", ts(type = "string"))] &'static str);
```

For type aliases (e.g. `type TurnNumber = u32`), annotate the field instead of the alias:

```rust
pub struct SessionMetadata {
    #[cfg_attr(feature = "typegen", ts(type = "number"))]
    pub turn_number: TurnNumber,
}
```

If the crate doesn't already opt into typegen, add to its `Cargo.toml`:

```toml
[features]
typegen = ["dep:ts-rs"]

[dependencies]
ts-rs = { workspace = true, optional = true }
```

Then propagate it up: `polaris_internal/Cargo.toml` adds `typegen = ["my_crate/typegen"]`, and the workspace root forwards through `polaris_internal/typegen`.

## CI Rule: Regenerate Must Be a No-Op

`cargo make test` runs with `--all-features`, which enables `typegen` and rewrites the `bindings/` directory. CI then runs `git diff --exit-code bindings/` тАФ if the regenerated output differs from what's checked in, the build fails.

Locally, after touching any `TS`-derived type:

```bash
cargo make gen-bindings
git add bindings/ts/src
# If you added a new type, also update bindings/ts/src/index.ts.
```

Skipping the regen-and-commit step is the most common cause of a red CI build for typegen-related PRs.
